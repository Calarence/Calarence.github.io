python源码剖析读书笔记一--python源码剖析

發表於 2017-06-26   |   分類於 读书笔记

###python总体架构：
如图python整体架构可分为三部分：

  • 最左边包括python提供的Modules、Library以及用户自定义的模块；
  • 右边部分是python运行时环境，包括对象/类型结构，内存分配器和运行时状态信息。运行时状态维护了解释器在执行字节码时不同的状态（比如正常状态和异常状态）之间切换的动作；内存分配器负责python创建对象时，内存的分配;而Object/Type包括了python中的各种内建对象，比如整数、list、dict，以及用户自定义的类型和对象
  • 中间部分是python核心。Scanner对应词法分析，将文件输入的Python源代码或从命令行输入的一行行python代码切分为一个个的token；parser对应语法分析，在scanner的分析结果上进行语法分析，建立抽象语法树；compiler根据建立的AST生成指令集合–python字节码；最后由Code Evaluator来执行这些字节码。

###python源码结构（以python2.5为例）

  • Include：该目录下包含了python提供的所有头文件，如果用户需要自己用C或C++来编写自定义模块拓展python，那么就需要用到这里提供的头文件。
  • Lib: 该目录下包好了python自带的所有标准库，Lib中的库都是用Python语言编写的。
  • Modules:该目录中包含了所有用C语言编写的模块，如random、cStringIO等。Modules中的模块都是那些对速度要求非常严格的模块，而有一些对速度没有太严格要求的模块，如os，就用python编写，放在Lib目录下
  • Parser: 该目录包括了python解释器中的Scanner和Parser，即对Python源代码进行词法分析和语法分析的部分。除此之外，Parser目录下还包含了一些有用的工具，这些工具能根据python语言的语法自动生成python语言的词法和语法分析器
  • Objects: 该目录中包含了所有python的内建对象，如整数、list、dict等。同时，该目录还包含了python在运行时需要的所有的内部使用对象的实现。
  • python：该目录下包含了python解释器中的Compiler和执行引擎部分，是python运行的核心所在

关于编译器与解释b性语言器：
将编程语言编译成机器码有两种传统方法：使用编译器（compiler）预先编译，或者使用解释器（interpreter）一边编译一边运行。

  • 编译器的工作方式是，通过编译程序直接将我们的源代码翻译成当前系统环境下CPU可执行的机器码，下次只需要执行这些翻译完成后的机器代码即可。
    这种方式的好处是执行效率非常高，因为我们最终生成的程序与直接用0、 1写的程序并没有 太大的区别。但是这种方式的缺点就是可移植性很差， 很显然不能把在 Windows 系统下编译 连接成的EXE文件拿到Linux系统中去执行，Xcode编译的 App也只能在iOS和OSX系统中运行。
  • 解释器的工作方式是将代码一句一句解释给CPU执行。解释器不会一次把整个程序翻译出来，而只像一位“中间人”，每次运行程序时都需要先转成另一种语言再运行，因此解释器的程序运行速度比较缓慢。它每翻译一行程序叙述就立刻运行，然后再翻译下 一行，再运行，如此不停地进行下去。这种方式的一种好处就是平台无关性， 我们只要写出 程序代码， 那么它在任何操作系统、任何环境中都可以执行， 当然代价就是极低的效率。 第二个优点是， 这个“编辑-解释-除错”的循环通常比“编辑-编译-运行-除错”的循环省时许多。

  • init(self):初始化对象，在创建新对象时调用,这个方法可以用来对你的对象做一些你希望的初始化

  • new(cls,*args,**kwd) 实例的生成操作.在init()之前被调用，用于生成实例对象。利用这个方法和类属性的特性可以实现设计模式中的单例模式，代码如下：

    class Singleton(object):
 __instance = None

    def __init__(self):
        pass

    def __new__(cls,*args,**kwd):
        if Singleton.__instance is None:
            Singleton.__instance = object.__new__(cls,*args,**kwd)
        return Singleton.__instance

if __name__ == '__main__':
s1 = Singleton()
s2 = Singleton()
print("s1 id %s ,s2 id %s " % (id(s1),id(s2)))
  • getattr(self,name) 获取属性的值，当读取对象的某个属性时，python会自动调用getattr()方法
  • setattr(self, name):设置属性值，当使用赋值语句对属性进行设置时，python会自动调用setattr()方法
  • getattribute(self, name)：与getattr(self,name)类似,如果定义了getattr(self, name),则先调用getattr。
  • getitem(self,key) 获取序列的索引key对应的值，等价于seq[key]，详细用法

  • str():str()用于表示对象代表的含义，返回一个字符串;默认输出对象，可以对其进行定制化，输出想要的东西。示例代码如下：

    class Fruit(object):
"""docstring for Fruit"""

    def __init__(self, color="red", price=0):
        self.color = color
        self.price = price

    def __str__(self):
        return self.__doc__

if __name__ == '__main__':
    f1 = Fruit("blue", 1)
    print(f1) # 输出docstring for Fruit

RESTful API最佳实践

REST

越来越多的公司让服务端所有商业逻辑都以RESTful API的方式暴露给客户端，浏览器用户可以使用Ajax、HTML5技术，通过HTTP的方式与后台直接交互。这种统一的机制，既减少了开发复杂度，又有非常好的扩展性。

REST:Representational State Transfer–表现层状态转化，这里指的是资源的表现层。资源是指web上一切可识别、可命名、可被找到并处理的实体。比如HTML页面、音频文件、图片等。用一个URL指向资源，使用HTTP方法操作资源。URI可进一步划分为统一资源名(URN)和统一资源定位符（URL），其中URL可以定位HTTP网址、FTP服务器和文件路径等，符合绝大多数场景，所以一般都可以用URL。

REST架构风格的5个约束：

1、客户端-服务器端。这种Client/Server架构形式提供了基本的分布式，客户端发起请求，服务端决定响应或拒绝请求，如果出错，则返回错误信息，由客户端处理异常。
2、无状态。通信的会话状态应该由客户端负责维护，也就是请求中包含了全部的必要信息。如果使用基于服务器端的会话，那么需要保证指定会话使用同一个服务器响应所有请求；要么得创建一个可供所有服务器访问的公用的会话存储区，对每个请求都额外访问这个集中式的数据存储区获得会话状态
3、缓存。无状态就表示可能出现重复的请求，事实上这些请求只需要第一次真正的执行，其余的请求都可以享用这个已完成的结果而直接响应，所以缓存可以抵消一部分无状态带来的响应
4、统一接口。统一接口意味着每个REST应用都共享一种通用架构。
5、分层系统：将系统划分为几个部分，每个部分负责一部分相对单一的职责，然后通过上下层的依赖和调用组成一个完整的系统。通常可以划分为如下三层：
应用层：负责返回JSON数据和其他业务逻辑
服务层: 为应用层提供服务支持
数据库访问层：提供数据访问和存储的服务，如数据库、缓存系统、文件系统、搜索引擎等

RESTful API设计指南：

  • 应该将API部署在专用域名之下：https://api.example.com
  • 不用大写
  • 用中杠-不用下杠_
  • 参数列表要encode
  • 使用名词来表示资源,URI中的名词表示资源集合，使用复数形式；
  • 关注请求头：一定要看请求头信息，并给予正确的状态码。举个例子，假设服务器端只能返回JSON格式，如果客户端的头信息的Accept字段要求返回application/xml，这个时候就不应该返回application/josn类型的数据，而应该返回406错误。
  • 合理使用请求方法和状态码：
方法 语义
OPTIONS 用于获取资源支持的所有HTTP方法
HEAD 用于只获取请求某个资源返回头信息
GET 用于从服务器获取某个资源的信息，1.完成请求后，返回状态码200 OK 2.完成请求后，需要返回被请求的资源详细信息
POST 用于创建新资源， 1、创建完成后，返回状态码201 Created 2、完成请求后，需要返回被创建的资源详细信息
PUT 用于完整的替换资源或创建指定身份的资源，如创建id为123的某个资源：1、如果是创建了资源，则返回201created 2.如果是替换了资源，则返回200 ok
PATCH 用于局部更新资源： 1、 完成请求后，返回状态码200 ok；2. 完成请求后，需要返回被修改的资源详细信息
DELETE 用于删除某个资源，完成请求后返回状态码204 NO Content

使用201 created响应时，应该带location，指向新建资源的地址
使用405 method not allowed响应时，应该带有allow头，告诉客户端对该资源有效的http方法

错误处理

就像HTML的出错页面向访问者展示了有用的错误消息一样，API也应该用之前熟悉易读的格式来提供有用的错误消息。错误的表现形式应该跟其他资源保持一致，只是用一些自己的字段。

API应该一直返回合理的HTTP状态码。API错误一般情况下分成两类：代表客户端错误的400系列状态码和代表服务端错误的500系列状态码。API至少把所有400系列错误统一用易读的JSON格式来展示。如果可能（比如，如果负载均衡和反向代理能够创建自定义错误内容的话），500系列的状态码也这么弄。

JSON错误内容应该为开发者提供一些东西 - 有用的错误消息，唯一的错误码（通过它可以在文档中找到更多错误细节），可能的话提供错误细节描述。用JSON格式来输出错误看起来这样：

{
  "code" : 1234,
  "message" : "Something bad happened :(",
  "description" : "More details about the error here"
}

对于PUT、PATCH和POST的请求进行的校验错误需要嵌套多个字段。最佳做法是用固定的错误码来表示校验失败，然后在额外的errors字段中提供错误的细节，像这样：

{
  "code" : 1024,
  "message" : "Validation Failed",
  "errors" : [
    {
      "code" : 5432,
      "field" : "first_name",
      "message" : "First name cannot have fancy characters"
    },
    {
       "code" : 5622,
       "field" : "password",
       "message" : "Password cannot be blank"
    }
  ]
}

对输出结果不再包装

不恰当的响应例子：

HTTP/1.1 200 OK
   {
   'success': true,
   'data': {'id':1,'name':'xiaoming'}
   }

应该直接返回data的数据

HTTP/1.1 200 OK
  {
  'id':1, 'name':'xiaoming'
  }

版本

保存在URI中， 如api/v2

URI失效和迁移

随着业务的发展，会出现一些API失效或迁移。对失效的API，应该返回“404 not found”或“404 gone”；对迁移的API ，返回301重定向

速度限制

为了避免请求泛滥，给API设置速度限制很重要。加入速度限制功能以后，应该提示用户，可以参照Github的返回头，如下所示：

X-RateLimit-Limit: 当前时间段允许的并发请求数
X-RateLimit-Remaining: 当前时间段保留的请求数
X-RateLimit-Reset: 当前时间段剩余的秒数

缓存

数据内容在一段时间内不会变动，这时我们就可以合理减少HTTP响应内容。应该在响应头中携带Last-Modified、ETag、Vary、Date等信息，客户端可以在随后请求这些资源时，在请求头中使用If-Modified-Since、If-None-Match等来确认资源是否经过修改。如果资源没有做过修改过，那么就响应“304 Not Modified”,并且不在响应实体中返回任何内容。

并发控制

缺少并发控制的PUT和PATCH请求可能导致“更新丢失”。这个时候可以使用Last-Modified和ETag头来实现条件请求。

身份认证

API的身份认证应该使用OAuth 2.0框架。

python编码：

计算机是以0-1的方式存储数据的，8比特（8个1）为一个字节，一个字节（11111111）所能表示的最大数字为255，两个字节（11111111 11111111）所能表示的最大数字为65535.
美国人最早研制计算机的时候，将128个字符（字母、数字以及特殊符号）保存到计算机里，用一个字节就够了。这种编码成为ASCII编码。
由于中文用一字节（128）表示不了，UnicodeDecodeError: 'ascii' codec can't decode byte 0xe4 in position 0: ordinal not in range(128) 所以会报这个错误。后来中文有了自己的编码–GB2312。
为了将所有语言统一编码，Unicode应运而生。
Unicode编码用两个字节表示一个字符，这样就会造成一定的浪费：像A完全可以用一个字节表示，第二个字节就浪费掉了。如果表示全英文，用Unicode编码比ASCII编码需要多一倍的存储空间，在存储和传输上就十分不划算。
为了节省空间，UTF-8编码出现了–它是可变长的Unicode编码。UTF-8编码把一个Unicode字符根据不同的数字大小编码成1-6个字节，常用的英文字母被编码成1个字节，汉字通常是3个字节，只有很生僻的字符才会被编码成4-6个字节。
ASCII编码实际上可以被看成是UTF-8编码的一部分，所以，大量只支持ASCII编码的历史遗留软件可以在UTF-8编码下继续工作。

在计算机内存中，统一使用Unicode编码，当需要保存到硬盘或者需要传输的时候，就转换为UTF-8编码.
由于Python的字符串类型是str，在内存中以Unicode表示，一个字符对应若干个字节。如果要在网络上传输，或者保存到磁盘上，就需要把str变为以字节为单位的bytes。
以Unicode表示的str通过encode()方法可以编码为指定的bytes

>>> 'ABC'.encode('ascii')
b'ABC'
>>> '中文'.encode('utf-8')
b'\xe4\xb8\xad\xe6\x96\x87'
>>> '中文'.encode('ascii')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
UnicodeEncodeError: 'ascii' codec can't encode characters in position 0-1: ordinal not in range(128)

纯英文的str可以用ASCII编码为bytes，内容是一样的，含有中文的str可以用UTF-8编码为bytes。含有中文的str无法用ASCII编码，因为中文编码的范围超过了ASCII编码的范围，Python会报错
如果我们从网络或磁盘上读取了字节流，那么读到的数据就是bytes。要把bytes变为str，就需要用decode()方法
str和bytes互相转换时，需要指定编码。最常用的编码是UTF-8

大纲

  2. 使用rdis-cli来访问server
  3. 在应用中使用Redis
  4. 更加正确地安装Redis
  5. redis的进一步学习

Redis的安装

推荐使用的Redis安装方式是源码编译安装，所需依赖项为GCC编译器和libc。注意Linux包管理器安装的版本可能不是最新版。
Redis最新稳定版：地址
安装步骤：

wget http://download.redis.io/redis-stable.tar.gz
tar xvzf redis-stable.tar.gz
cd redis-stable
make
sudo make install

启动redis：

redis-server

这个命令在测试环境下可以这样用，在生产环境下，要使用 redis-server /etc/redis.conf，可以用redis源码根目录下的redis.conf作为模板，来写自己的配置文件。

检查Redis是否正常工作

外部程序使用TCP协议和Redis特殊协议同Redis通信。这个特殊协议在Redis client 库中实现，适用于各种语言。为了更简单的使用Redis，Redis提供了命令行工具，来向Redis发送命令。这个工具就是redis-cli。
要检查Redis是否正常工作的第一件事情就是使用redis-cli发送PING命令。

$ redis-cli ping
PONG

redis-cli后的命令以及参数将会发送给localhost，端口为6379的Redis实例。可以通过redis-cli来修改host和port。

另一种运行redis-cli的方式是不带参数：这样将会进入交互模式，可以输入不同的命令来观察回复：

$ redis-cli                                                                
redis 127.0.0.1:6379> ping
PONG
redis 127.0.0.1:6379> set mykey somevalue
OK
redis 127.0.0.1:6379> get mykey
"somevalue"

做到这一步，就能跟Redis通信了。接下来学习Redis的数据类型

centos7下使用nginx+uwsgi部署django项目

發表於 2017-03-03

生产环境搭建：

  1. nginx安装：yum install -y nginx,启动nginx：nginx,访问本机ip地址即可看到nginx欢迎界面
  2. mysql的安装和配置: 链接
  3. virtualenv安装和使用：
    pip install virtualenv
    创建虚拟环境testEnv（在当前目录创建，也可指定其他目录） virtualenv testEnv
    进入虚机环境 source testEnv/bin/activate
    退出虚拟环境 deactivate
  4. 导出项目依赖项：pip freeze > requirements.txt
  5. 虚拟环境中安装依赖项（将第四步导出的txt导入centos中）:pip install -r requirements.txt若某些依赖项安装较慢，可使用源
    pip源设置： vi ~/.pip/pip.conf 内容：
    [global] trusted-host=mirrors.aliyun.com index-url = http://mirrors.aliyun.com/pypi/simple
  6. 测试项目可正常运行 python manage.py runserver
  7. 安装uwsgi：pip install uwsgi,如果报缺少编译器的错误,则安装基础开发包yum groupinstall "Development tools" , yum install -y python-devel
  8. 测试uwsgi可启动django项目:uwsgi --http :8000 --wsgi-file wsgi.py或uwsgi --http :8000 --module yourporject.wsgi注意两个命令执行时的路径，没有错误可正常访问django应用
  9. 重启uwsgi pkill -f uwsgi

  10. nginx配置：新建nginx.conf,添加以下内容：
    `

       # the upstream component nginx needs to connect to
   upstream django {
   # server unix:///path/to/your/mysite/mysite.sock; # for a file socket
   server 127.0.0.1:8000; # for a web port socket (we'll use this first)
   }   
   # configuration of the server
   server {
   # the port your site will be served on
   listen      80;
   # the domain name it will serve for
   server_name 你的ip地址 ; # substitute your machine's IP address or FQDN
   charset     utf-8;
   # max upload size
   client_max_body_size 75M;   # adjust to taste
   # Django media
   location /media  {
   alias 你的目录/yourproject/media;  # 指向django的media目录
   }
   location /static {
   alias 你的目录/yourproject/static; # 指向django的static目录
   }
   # Finally, send all non-media requests to the Django server.
  location / {
  uwsgi_pass  django;
  include     uwsgi_params; # the uwsgi_params file you installed
   }
}

    将该配置文件copy到/etc/nginx/conf.d或sudo ln -s 你的目录/yourproject/conf/nginx/uc_nginx.conf /etc/nginx/conf.d/`

  11. 拉取所有需要的static file 到同一个目录:
    settings.py中添加STATIC_ROOT = os.path.join(BASE_DIR, "static/")
    运行python manage.py collectstatic

  12. 重启nginx:nginx

  13. 通过配置文件启动uwsgi:新建uwsgi.ini 配置文件，内容如下：
    `

    # mysite_uwsgi.ini file
[uwsgi]
# Django-related settings
# the base directory (full path)
chdir           = 你的目录/yourproject
# Django's wsgi file
module          = yourproject.wsgi
# the virtualenv (full path)
# process-related settings
# master
master          = true
# maximum number of worker processes
processes       = 10
# the socket (use the full path to be safe
socket          = 127.0.0.1:8000
# ... with appropriate permissions - may be needed
# chmod-socket    = 664
# clear environment on exit
vacuum          = true
virtualenv = 你的目录/testEnv`

    注意:

    `

chdir： 表示需要操作的目录，也就是项目的目录
module： wsgi文件的路径
processes： 进程数
virtualenv：虚拟环境的目录
`
  14. 启动uwsgi：uwsgi -i 你的目录/uwsgi.ini &
  15. 访问：http://你的ip地址/
  16. 可能遇到的问题：重启nginx或uwsgi时会有端口占用，执行killall -9 uwsgi（或nginx）

centos7下mysql的安装和配置

下载安装mysql-server:

  1. wget http://dev.mysql.com/get/mysql-community-release-el7-5.noarch.rpm
  2. rpm -ivh mysql-community-release-el7-5.noarch.rpm
  3. yum install mysql-community-server

安装完成后重启mysql服务
service mysqld restart

登录mysql(初次登录没有密码)
mysql -u root

设置密码
set password for 'root'@'localhost' =password('password');

再次登录
mysql -u root -p,会提示输入密码，输入密码后，回车即可进入数据库

配置mysql

  1. 编码，编辑/etc/my.cnf，最后添加
    [mysql]
    default-character-set =utf8
    这里的字符编码必须和/usr/share/mysql/charsets/Index.xml中一致

  2. 远程连接
    把在所有数据库的所有表的所有权限赋值给位于所有IP地址的root用户：
    grant all privileges on *.* to root@'%'identified by 'password';
    FLUSH PRIVILEGES
