spring依赖注入示例之hello world

写在前面

任何一个应用都会有两个或两个以上的类组成，这些类相互协作以完成特定的逻辑，这期间会出现高度耦合的代码。而依赖注入的出现，可以让我们比较容易的实现松耦合。通过依赖注入，对象的依赖关系由系统中负责协调各对象的第三方组件在创建对象的时候进行设定。对象无需创建或管理它们的依赖关系，而依赖关系会自动被注入到需要它们的对象当中去。

hello world

  • 情景：有一个类，它负责根据时间来输出hello world，如早上输出“hello world in morning”,晚上输出“hello world at night”。
  • 设计：Interface–HelloWorldService定义一个接口sayHelloWorld(),而HelloWorldA和HelloWorldB负责具体的实现；类OutPutHelper实现具体的“业务”。
  • 步骤：
    • 新建maven工程，用来管理Spring jar包依赖
    • 在pom文件中引入spring依赖
    • 代码实现如下：
        //HelloWorldService接口定义
        public interface HelloWorldService {
        String sayHelloWorld();
        }
        //HelloWorldA的实现
        public class HelloWorldA implements HelloWorldService {
        public String sayHelloWorld() {
        return "Hello World in  morning";
        }
        }
        //HelloWorldB的实现
        public class HelloWorldB implements HelloWorldService {
        public String sayHelloWorld() {
        return "Hello World in night";
        }
        }
        //输出类实现
        public class OutPutHelper {
        private HelloWorldService helloWorldService;

         public OutPutHelper(HelloWorldService  helloWorldService){
        this.helloWorldService = helloWorldService;
        }

         public void outPut(){
         System.out.println(helloWorldService.sayHelloWorld());
        }
        }
* spring配置文件

        <bean id="helloWorldA" class="com.appleieye.service.impl.HelloWorldA" />
         <bean id="helloWorldB"  class="com.appleieye.service.impl.HelloWorldB" />
        <bean id="outPutHelper" class="com.appleieye.outputer.OutPutHelper" >
        <constructor-arg ref="helloWorldA" />
        </bean>


* 测试


           public class Main {
           public static void main(String[] args) {
           ClassPathXmlApplicationContext context = new ClassPathXmlApplicationContext("spring.xml");
            OutPutHelper outPutHelper = (OutPutHelper) context.getBean("outPutHelper");
           outPutHelper.outPut();

             }
          }

如何正确的重写equals方法

写在前面

Object类是Java中所有类的始祖。Object类的设计是为了扩展，其中它所有的非final方法(如equal/hashcode/toString/clone/finalize)都是可以被覆盖的。本文将介绍如何以正确的姿势覆盖equals和hashcode方法。

为什么要重写equals方法？

先来看一下equals方法在Object类中的实现：

public boolean equals(Object obj) {
        return (this == obj);
    }

可见，equals方法的默认实现是判断两个引用是否相等。这种方式是判断两个对象的引用。如果引用相同，则认为两个对象是相等的，这一点毋庸置疑。但是它欠缺了一点：如果两个对象的引用不同，但是对象的各种状态相同，也认为这两个对象相等。举个例子，把引用比作为家庭住址，住的人可能因搬家导致家庭住址发生变化，但是人的各种信息，如姓名/性别/外貌/手机号/生份证号等，如果相同，则认为是两个人是相等的。

如何正确的重写equals方法：
假设有一个person类，它所拥有的属性有name、sex、age，那么该如何重写它的equals方法呢？
过程如下：

  • 判断this与ohterObject的引用是否相同
  • 判断otherObject是否为null
  • 判断this与otherObject是否属于同一个类
  • 将otherObject转换为相应的类类型变量
  • 对所有属性进行比对，使用==比较基本类型，使用equals比较对象类型，若所有的属性都匹配，则返回true，否则返回false。

代码如下：

import java.util.Objects;

public class Person {

private String name;
private String sex;
private String address;
private int age;

public Person(String name, String sex, String address, int age) {
    this.name = name;
    this.sex = sex;
    this.address = address;
    this.age = age;
}

@Override
public boolean equals(Object obj) {
    if (this == obj) return true;
    if (obj == null) return false;
    if (getClass()!=obj.getClass()) return false;
    Person other = (Person) obj;
    return Objects.equals(name,other.name)&&Objects.equals(sex,other.sex)
            &&Objects.equals(address,other.address)&&age == other.age;
}
}

重写hashCode方法

在重新定义equal方法后，必须重新定义hashCode方法。因为equals方法与hashCode的定义必须一致：如果x.equals(y)返回true，则x和y必须有相同的hashcode值，这是因为hashtable的原因，具体可去看hashtable的实现。
hashCode方法重写如下：

@Override
    public int hashCode() {
        return Objects.hash(name,sex,address,age);
    }

Java中锁的实现原理

  • 原理
    Lock接口的实现是通过聚合了一个队列同步器，利用同步器实现锁的语义。或者可以理解为Lock接口是通过队列同步器实现的。
  • 队列同步器的使用方式
    同步器的主要使用方式是继承，使用者需要继承同步器并重写指定的方法，随后将同步器组合在自定义同步组件的实现中，并调用同步器的模版方法，而这些模版方法将会调用使用者重写的方法。
  • 队列同步器的API或模版方法
  • 队列同步器的实现分析
    • 同步队列：同步器依赖内部的同步队列，FIFO双向队列来完成同步状态的管理，当前线程获取同步状态失败时，同步器会将当前线程以及等待状态信息构造成为一个节点，将其加入队列尾部，并阻塞当前线程；当同步状态释放时，会把首节点中的线程唤醒，使其再次尝试获取同步状态。
    • 入队：当一个线程成功获取了同步状态后，其他线程无法获取到同步状态，转而被构造成为节点，添加到尾部，然而加入到队列的过程必须要保证线程安全，因此同步器提供了一个基于CAS的设置尾节点的方法：compareAndSetTail(Node expect,Node uodate),它需要传递当前线程认为的尾节点和当前节点，只有设置成功后，当前节点才正式与之前的尾节点建立关联。
    • 出队：首节点的线程在释放同步状态时，将会唤醒后继节点，而后继节点将会在获取同步状态成功时将自己设为首节点，因为只有一个线程能够获取到同步状态，因此不需要CAS来设置。
    • 独占式同步状态获取与释放
      • 获取流程如下图所示
      • 释放：释放同步状态，并唤醒下一个节点
    • 共享式同步状态获取与释放
      • 获取：与独占式的区别：共享式创建的Node的nextWaiter是一个SHARED常量来标志，而独占式标志是一个NULL节点；在队列等待时，独占式如果改变Status成功则就直接设置为头结点就返回了，但是共享式会进行判断，如果是SHARED并且Status>=0则会设置头结点并且直接将下一个节点也unpark唤醒以便传递下去；
      • 释放：因为释放同步状态的操作会同时来自多个线程，所以一般通过循环和CAS来保证释放。

类、超类和子类

子类构造器：

使用super调用构造器的语句必须是子类构造器的第一条语句。

  • this的作用：一是引用隐式参数；二是调用该类其他的构造器
  • super的作用:一是调用超类的方法；二是调用超类的构造器

多态

程序中出现超类对象的任何地方都可以用子类对象替换。一个变量既可以引用自己类的对象，也可以引用其任何一个子类的对象。
不能将超类的引用赋给子类变量

方法调用

强制类型转换：

将一个子类的引用赋给一个超类变量，编译器是允许的；但将一个超类的引用赋给一个子类变量，必须进行类型转换。
在进行类型转换时，需要注意以下两点：

  • 只能在继承层次内进行类型转换
  • 在将超类转换成子类之前，应该使用instancef进行检查

抽象类：

  • 包含一个或多个抽象方法
  • 可以包含具体数据和具体方法
  • 抽象类不能被实例化，但可以定义一个抽象类的对象变量，只能引用非抽象子类的对象

4个访问修饰符

  • private：仅本类可见
  • public：对所有类可见
  • protected:对本包和所有子类可见
  • default：对本包可见

Object：所有类的超类

1、equals方法：在Object类中，equals用于判断两个对象是否有相同的引用；如果两个对象具有相同的引用，则两者一定是相同的。然而，经常需要检测两个对象状态的相等性，如果两个对象的状态相同，则认为两个对象是相等的

编写合格的equals方法：

  • 参数命名为otherObject，稍后将其转换成另一个叫做other的变量
  • 检测this与otherObject是否引用同一个对象
    if(this==otherObject) return true
  • 检测otherObject是否为null
    if(otherObject==null) return false
  • 比较this与otherObject是否属于同一个类。如果equals的语义在每个子类中有所改变，就使用getClass检测：
    if(getClass()!=otherObject.getClass()) return false;
  • 如果所有子类拥有统一的语义，则使用instanceof检测：
    if(!(otherObject instanceof ClassName)) return false;
  • 如果在子类中重新定义了equals，就要在其中包含调用super.equals(otherObject)
  • 将ohterObject转换为相应的类的类型变量
    ClassName other = (ClassName)otherObject;
  • 对所有需要比较的域进行比较。使用==比较基本类型域，使用equals比较对象域；如果所有的域都匹配，就返回true；否则返回false
    return field1 == other.field1 && Objects.equals(field2,other.field2)

2、hashCode方法
hash code是由对象导出的一个整形值。散列码是没有规律的，如果x和y是不同的对象，x.hashCode()与y.hashCode()基本不会相同。
Equals方法与hashCode的定义必须保持一致。如果x.equals(y)返回true，那么x.hashCode()与y.hashCode()必须具有相同的值。例如，如果在equals方法中用name这个属性去做判断，那么hashCode方法就需要散列name，而不是其他属性。

3、toString方法：用于返回标识对象的字符串值，形式:类的名字，随后是一对方括号括起来的域值

import java.util.Objects;
public class Employee {

private String name;
private double salary;



public Employee(String name, double salary) {
    this.name = name;
    this.salary = salary;
}

public String getName(){
    return name;
}

public double getSalary(){
    return salary;
}

public void raiseSalry(double byPercent){
    double raise = salary * byPercent / 100;
    salary += raise;
}

public boolean equals(Object otherObject){
    if(this == otherObject) return true;
    if(otherObject == null) return  false;
    if(getClass()!=otherObject.getClass()) return false;
    Employee other = (Employee) otherObject;
    return Objects.equals(name, other.name) && salary == other.salary;

    return Objects.hash(name,salary);
}

public String toString(){

}
}

Java中锁的基本概念

發表於 2018-03-06   |   分類於 blog

java中锁的基本知识

  • 为什么有了synchronized还需要Lock
    Lock接口出现在synchronized关键字之后，既然synchronized提供了锁的功能，那么为什么还需要Lock接口呢？原因是Lock接口提供了synchronized所不具备的同步特性。
    Lock缺少了synchronized隐式获取释放锁的便捷性，但却拥有了锁获取与释放的可操作性。例如，针对一个场景，先获得锁A，再获得锁B，在获得锁B后，释放锁A同时获取锁C，当锁C获得后，再释放锁B同时获取锁D，依次类推，这种场景下，用synchronized就不容易实现。此外，Lock接口还拥有可中断的获取锁、超时获取锁、非阻塞地获取锁等多种synchronized关键字所不具备的特性。
  • Lock的API（如下）
    | 方法名称 | 描述 |
    | ——– | —–: | :—-: |
    | void lock() |获取锁，调用该方法当前线程将会获取锁，当锁获得后，从该方法返回 |
    | boolean tryLock | 尝试非阻塞的获取锁，调用该方法后立刻返回，如果能够获取则返回true，否则返回false | |
    | boolean tryLock(long time,TimeUnit unit)throws InterruptedException |超时的获取锁，当前线程在以下3种情况下会返回：1、当前线程在超时时间内获得了锁 2、当前线程在超时时间内被中断 3、超时时间结束，返回false| |
    |void lockInterruptibly() throws InterruptedException|可中断地获取锁，和lock()方法不同之处在于该方法会响应中断，即在锁的获取中可以中断当前线程|
    |void unlock()|释放锁|
    |Condition new Condition()|获取等待通知组件，该组件和当前的锁绑定，当前线程只有获得了锁，才能调用该组件的wait()方法，而调用后，当前线程将释放锁|
  • Lock的使用方式（代码如下）
/*
*注意获取锁与释放锁的位置
*获取锁的过程没写在try块中，原因是如果在获取锁时发生了异常，异常抛出的同时，也会导致锁释放
*在finally块中释放锁，目的是保证在获取到锁之后，最终能够被释放。
**/
Lock lock = new ReentrantLock();
lock.lock();
try{
}finally{
lock.unlock();
}

在synchronized与Lock之间进行选择

Lock接口在锁操作上提供的语义与synchronized相同，不过除此之外，Lock接口还提供了其他特性：可中断获取锁，非阻塞获取锁，超时获取锁等；但在使用Lock接口时会有很大的风险，如忘记在finally块中调用unlock，虽然代码表面上可以运行，但实际上已经埋下了一颗定时炸弹，并且很有可能伤及其他代码。仅当synchronized不能满足需求时，才考虑使用Lock。这些场景包括可定时的、可轮询的与可中断的锁获取操作、公平队列、以及非机构的锁。否则，应该是优先使用synchronized。

java内存分配与回收策略

發表於 2018-01-05   |   分類於 blog

java内存分配与回收策略

  1. 对象优先在Eden分配：大多数情况下，对象在新生代Eden区中分配。当Eden区没有足够的空间时，虚拟机将发起一次Minor GC。如果GC时发现已有对象无法放入Survivor空间，则通过分配担保机制提前转移到老年代中去。
  2. 大对象直接进入老年代。大对象是指需要大量连续内存空间的java对象，最典型的大对象如很长的字符串以及数组。虚拟机提供了一个-XX：PretenureSizeThresold参数，令大于该值的对象直接才老年代分配
  3. 长期存活的对象进入老年代：虚拟机给每个对象定义了一个对象年龄计数器。通过该计数器来判断是否需要进入老年代。如果对象在Eden出生并且经过第一次Minor GC仍然能够存活下来，并且能被survivor容纳，将被移动到survivor空间中，并且年龄设为1。对象在Survivor区中每熬过一次Minor GC，年龄就增长一岁。当它的年龄增加到一定程度，默认为15岁，就会被移动到老年代中。对象晋升老年代的年龄阀值，可以通过参数-XX：MaxTenuringThresold来设置。
  4. 动态对象年龄判定：为了能更好地适应不同程序的内存状况，虚拟机并不是永远地要求对象的年龄必须达到了MaxTenuringThresold才能晋升老年代。如果在Survivor空间中，相同年龄所有对象的大小的总和大于Survivor空间的一半，年龄大于或等于该年龄的对象就可以直接进入到老年代中，无须等到MaxTenuringThresold所要求的年龄。
  5. 空间分配担保：在Minor GC之前，虚拟机会检查老年代最大可用的连续空间是否大于新生代所有对象总空间，如果这个条件成立，那么Minor GC可以确保是安全的。如果不成立，则查看HandlePromotionFailure设置是否允许担保失败。如果允许，那么会继续检查老年代最大可用连续空间是否大于历次晋升到老年代对象的平均大小，如果大于，则会尝试进行Minor GC；如果小于或者不允许担保失败，则进行一次Full GC。大部分情况下会打开HandlePromotionFailure，以避免Full GC过于频繁。

java垃圾收集

發表於 2018-01-05   |   分類於 blog

java GC

GC需要考虑的三个问题：

  1. 哪些内存需要回收
  2. 何时回收
  3. 如何回收

对这三个问题的解决，也是本文的叙述顺序

1、哪些内存需要回收–可达分析算法来判断：
由于引用计数法无法解决循环引用的问题，所以在java中没有采用该方法，而是采用了一种叫做“可达分析”的算法。
这个算法的基本思路是通过一些列称为“GC Roots”的对象作为起始点，从这些起始点开始向下搜索，搜索走过的路径称为引用链。当一个对象到GC Roots没有任何引用链，也就是从GC Roots到这个对象不可达，则证明此对象时不可用的，由此判定这些对象是可回收的对象。
可作为GC Roots的对象包括下面几种：

  • 虚拟机栈中（本地变量表）引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中引用的对象

2、何时回收–安全点、安全区域才进行GC：
程序执行时并非在所有的地方都能停下来开始GC，只有到达安全点时才能暂停。
3、如何回收：回收算法（理论）+垃圾收集器（实现）：
回收算法：
> 标记清除：
算法分为两个阶段：标记（可达分析算法）和清除。它有两个不足：一个是效率问题，标记和清除这两个过程效率都不高；第二个是空间问题，会产生空间碎片。空间碎片太多可能会导致以后分配大对象时，无法找到足够内存而不得不提前触发另一次垃圾收集动作。
> 复制：将内存一分为二，每次只使用其中一块。当一块用完后，就将上面还存活的对象复制到另一块上，然后将使用过的内存一次性的回收。
优点：实现简单，运行高效
缺点：内存缩小为原来的一半，代价高昂
用于回收新生代对象
> 标记整理：标记阶段与‘标记-清除’方法一样，但后续步骤不是直接对可回收对象进行清理，而是让所有存活的对象都移动到一端，然后直接清理掉端边界以外的内存。
> 分代收集算法：将java堆分为新生代和老年代，新生代采用复制算法，而老年代采用标记-清除或标记整理算法。

垃圾收集器:

  • Serial收集器：
    最基本的、发展历史最悠久的新生代收集器；单线程，工作时会暂停其他所有的工作线程；虚拟机运行在client模式下的默认新生代收集器
  • ParNew收集器：
    Serial收集器的多线程版本；运行在server模式下虚拟机首选的新生代收集器；除Serial外，唯一的能与CMS收集器配合工作的收集器
  • Parallel Scavenge收集器：
    新生代收集器；并行多线程；与其他收集器关注用户线程停顿时间不一样，Parallel收集器关注的是吞吐量，吞吐量优先的收集器；Parallel Scavenage收集器提供了两个参数用于精确控制吞吐量，分别是控制最大停顿时间-XX:MaxGCPauseMillis参数以及设置吞吐量大小的-XX:GCTimeRatio；此外，还有一个参数-XX:+UseAdaptiveSizePolicy,这是一个开关参数，当这个参数打开后，就不需要手动指定新生代的大小（-Xmn），Eden与Survivor的比例(-XSS:SurvvivorRatio)、晋升老年代对象大小（-XX：PretenureSizeThreshold）等细节参数了，虚拟机会根据当前系统的运行情况收集性能监控信息，动态调整这些参数以提供最合适的停顿时间或者最大吞吐量，这种调节方式称为GC自适应的调节策略。
  • Serial Old收集器：Serial收集器的老年代版本；使用标记-整理算法；clint模式下的虚拟机使用
  • Parallel Old收集器：Parallel Scavenage收集器的老年代版本。
  • CMS（Concurrent Mark Sweep）收集器:并发标记、低停顿，以获取最短停顿时间为目标，采用标记-清除算法；服务端；老年代。
    运行过程：
  • 初始标记：仅标记GC Roots能直接关联到的对象，暂停用户线程
  • 并发标记：从GC Roots开始对堆中对象进行可达性分析，找出存活的对象，此过程时间较长，但可与用户线程并发执行
  • 重新标记：修正并发标记期间因用户程序继续运作而导致标记产生变动的那一部分对象的标记记录。
  • 并发清除：与用户线程并发执行
    三个缺点：
  • CMS收集器对CPU资源非常敏感
  • CMS收集器无法处理浮动的垃圾
  • 基于标记-清除，会产生空间碎片
  • G1收集器：面向服务端；并发与并行；分代收集；空间整合（使用标记-整理）；可预测的停顿，除了追求低停顿外，还能建立可预测的停顿时间模型，能让使用者指定在一个长度为M毫秒的时间片段内，消耗在垃圾收集上的时间不超过N毫秒；
    使用G1收集器时，Java堆的内存布局就与其他收集器有很大差别，它将整个java堆划分为多个大小相等的独立区域（Region）,虽然还保存着新生代与老年代，但他们不再是物理隔离的了，他们都是一部分Region的集合。
G1收集器之所以能建立可预测的停顿时间模型，是因为它可以跟踪各个Region里面的垃圾堆积的价值大小（回收所获得的的空间大小以及回收所需时间的经验值），在后台维护一个优先列表，每次根据允许的收集时间，优先回收价值最大的Region
过程：前三个过程与CMS收集器过程基本类似
