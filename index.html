RabbitMQ通信过程分析

总体流程概述：

生产者创建消息，并将消息投递给RabbitMq,具体来说，应该是RabbitMQ的交换器(Exchange),然后交换器根据规则(routing key),决定将消息投递到对应的队列(queues),这时候，监听队列的消费者就可以获取到消息了。

关键词：交换器、队列、绑定、路由键

具体流程如下：

  1. 生产者建立到RabbitMQ的TCP连接。在TCP的基础上，创建信道。信道是在建立在“真实的”TCP连接内的虚拟连接。不论是发布消息、订阅队列或是接收消息，这些动作都是通过信道完成的。可以把TCP想象成为一束光纤电缆，而信道是电缆中每条光纤。
  2. 生产者发送消息。消息包括两部分内容：有效荷载和标签。有效荷载是生产者要发送的数据，而标签对有效荷载的描述，它决定了谁将获得这些消息，具体内容包括交换器名称、路由键名称、编码、content-type、数据类型(如字符串还是自定义的类型)、delivery_mode等等。可以在管理界面—queues—get message中查看。
  3. 消息路由。路由器实际上只是一张查询表，表中记录了路由模式列表和匹配消息应发往的队列进程ID列表。当发布的消息匹配了交换器中的绑定规则时，实际上是由信道完成了匹配工作，并在匹配后建立到队列PID的连接，然后将消息传送过去。
  4. 队列对消息的处理。如果至少有一个消费者订阅了队列的话，消息会立即发送给订阅的消费者。如果消息到了无人订阅的队列，消息会在队列中等待，一旦有消费者订阅到该队列，队列上的消息就会发送给订阅者；当队列有多个消费者时，队列将消息以循环(round-robin)的方式发送给消费者；
  5. 消费者接收消息。消费者通过两种方式从队列中接收消息：同步(basic.get)和异步(basic.consume)。basic.get只能接收单条消息，如果想要获得更多的消息，需要再次发送basic.get命令，或者放在一个循环里，但是这样的性能不好，建议使用basic.consume。basic.consume模式，会在消息到达队列后，消费者自动接收消息，提高吞吐量。
  6. 消费者对消息的确认。消费者可以选择在接收到消息后对消息进行确认，这样是为了保证可靠性，但会带来性能上的损耗。可以设置auto_ack为true，一旦消费者接收到消息后，RabbitMQ就自动确认该消息；也可以手动进行确认。如果应用程序崩溃了导致没有确认消息，Rabbit会将消息发送给其他消费者；另一方面，如果没有确认消息的话，Rabbit将不会给该消费者发送更多的消息了，这是因为在上一条消息确认前，Rabbit认为这个消费者还没有准备好接收下一条消息。如果消费者想拒绝消息的话，可以使用reject命令。如果reject命令的requeue参数为true，RabbitMQ会将消息发送给下一个订阅的消费者；如果设为false的话，会将消息从队列中移除。

RabbitMQ常用命令

發表於 2018-05-26   |   分類於 blog

RabbitMQ常用命令

  • 启动节点：rabbitmq-server
  • 后台守护进程的方式启动：rabbitmq-server -detached
  • 停止节点：rabbitmqctl stop
  • 停止指定的某个节点：rabbitmqctl stop -n rabbit@[hostname]
  • 停止应用程序 rabbitmqctl stop_app
  • 启动应用程序 rabbitmqctl start_app
  • 启用management插件：rabbitmq-plugins enable rabbitmq_management
  • 列出队列和消息条目：rabbitmqctl list_queues，如果想列出某个vhost上的，后面加“-p hostname”
  • 列出exchange：rabbitmqctl list_exchanges，其中没有名称的direct类型的是默认的交换器
  • 列出绑定：rabbitmqctl list_bindings
  • 添加用户: rabbitmqctl add_user username password
  • 删除用户：rabbitmqctl delete_user
  • 列出所有用户：rabbitmqctl list_users
  • 修改密码 rabbitmqctl change_password username newpassword
  • 设置权限 rabbitmqctl set_permissions -p vhostname username 配置权限 写权限 读权限
  • 列出某vhost上的权限：rabbitmqctl list_permissions -p vhostname
  • 删除某vhost上的某用户权限： rabbitmqctl clear_permissions -p vhostname username

Spring与AMQP

  1. amqp工作流程：

消息的生产者将信息发布到一个exchage。exchange会根据某些绑定的一个或多个队列上，并将消息路由到队列上。消息的消费者会从队列中提取数据并进行处理。

2.四种exchage
> direct：如果消息的routing key与binding的routing key直接匹配的化，会将消息路由到该队列上
> topic:如果消息的routing key与binding的routing key符合通配符匹配的话，消息将会路由到该队列上
> headers:如果消息参数表中的头信息和值都与bingding参数表中的相匹配，消息将会路由到该队列上(header和direct完全一致，但是性能差很多，现在几乎不用了)
> fanout:不管消息的routing key和参数表的头信息/值是什么，消息将会路由到所有队列上。

3.amqp的一种实现–rabbitmq

  • 配置spring支持rabbitmq

springMVC示例之hello world

Spring MVC对请求的处理流程

流程图如下：

  1. DispatcherServlet称为前端控制器，它根据处理器映射将请求发送给对应的控制器
  2. 处理器映射：记录请求和它对应的控制器
  3. 请求的实际处理者。它将业务逻辑委托给一个或多个服务对象进行处理，产生信息(model)和视图(view),
  4. 控制器将模型数据打包，标注视图名，然后将请求连同模型和数据发送回DispatcherServlet.
  5. 视图解析器将视图名匹配为一个真正的视图，可能是jsp或html
  6. 将模型数据渲染到视图
  7. 返回响应

MVC的核心思想是业务数据抽取与业务数据呈现相分离

springMVC hello world实现

  1. 新建maven spring webapp工程，命名为springMVC
  2. 在pom中添加相关spring依赖
  3. 在web.xml中配置DispatcherServlet

简单理解该配置文件：

  • Spring context:由DispatcherServlet创建，用于加载web组件，如控制器、视图解析器和处理器映射。
  • Application context:由ContextLoaderListener创建，用来控制spring全局的特性,它是执行的是spring的上下文，用户加载bean，service，dao，数据库等配置信息

spring面向切面编程示例之打印日志

切面的概念

在[spring依赖注入示例之hello world][1]这篇文章中，实现了hello world的输出，这部分的逻辑可以说是“业务逻辑”。在实现业务逻辑时，我们会需要一些额外的操作，如通过打印日志来记录程序的执行情况(在输出hello world的前后都打印日志)；通过安全服务来判断用户是否登陆(只有登陆的用户才可以打印日志)；这些模块或服务可能是所有业务逻辑都用的到的，称之为横切关注点。

在知道了横切关注点的概念后，接下来考虑如何将其应用到业务逻辑中。一般思路会是在业务逻辑中调用这些服务，但这样会有两个问题：第一个是这些横切关注点的代码会重复出现在所有的业务逻辑代码中；第二个问题是当横切关注点代码发生改变时，所有的业务代码也需要跟着变化。而spring的面向切面编程很好的解决了以上两个问题，它能够使这些服务模块化，并以声明的方式将它们应用到它们需要影响的组件中去。

代码实现

  • pom中添加依赖

    <dependency>
 <groupId>org.aspectj</groupId>
 <artifactId>aspectjweaver</artifactId>
 <version>1.8.13</version>
</dependency>

  • 添加切面的实现

public class LogHelper {
public LogHelper() {
    }
public void sayBefore(){
    System.out.println("before saying hello world");
    }
public void sayAfter(){
    System.out.println("after saying hello world");
        }
      }

  • 配置切面

    <bean id="logHelper" class="com.appleieye.outputer.LogHelper">
</bean>
<aop:config>
<aop:aspect ref="logHelper">
<!--定义切点-->
<aop:pointcut id="helloWorld" expression="execution(* *.outPut())" />
<!--定义前置通知-->
<aop:before pointcut-ref="helloWorld" method="sayBefore"  />
<!--定义后置通知-->
<aop:after pointcut-ref="helloWorld" method="sayAfter"  />
</aop:aspect>
</aop:config>

这里使用了spirng的aop配置命名空间把logHelper声明为一个切面。首先，需要把logHelper声明为一个bean，然后在<aop:aspenct>元素中引用该bean。<aop:before>在outPut()方法前执行调用，而<aop:after>在outPut()方法后执行调用。
通过这种方式，使得logHelper被应用到outHelper中去，更重要的是，并没有显示的在outHeper中调用logHelper.

spring bean装配的三种方式之自动装配

發表於 2018-04-22   |   分類於 blog

spring bean装配的三种方式

写在前面

在之前的两个例子中，通过xml方式配置了依赖的对象，并由spring自动创建并注入依赖的对象，我们并没有在代码中额外的去增加紧耦合的代码。创建应用对象之间协作关系的行为通常称为装配。这也是依赖注入的本质。

bean装配的三种方式：

  • 在xml中进行显示配置
  • 在java中进行显示配置
  • 隐式的bean发现和自动装配

推荐顺序：优先使用自动配置，显式配置越少越好；如果非用显示配置，java代码配置要优先于xml配置；最后是xml配置

场景

考虑一个音箱系统(播放古老cd),由两部分组成：播放器(CDPlayer)和CD(compact Disc)。首先，要创建CompactDisc类，然后创建一个CDPlayer类，让spring发现它，并将CompactDisc bean注入进来。

两个接口

package com.appleieye.service;
public interface CompactDisc {
void play();
}

package com.appleieye.service;
public interface MediaPlayer {
void play();
}

自动化装配bean

spring从两个角度来实现自动装配：
> 组件扫描：spring会自动发现应用上下文中所创建的bean
> 自动装配：spring自动满足bean之间的依赖

  1. 组件扫描

    • 创建bean
    • 启动组件扫描

    代码如下：

    //创建bean
package com.appleieye.maya.api.impl;

import com.appleieye.maya.api.service.CompasticDisc;
import org.springframework.stereotype.Component;

@Component
public class SgtPeppers implements CompasticDisc {

private String title = "Sgt. Pepper' s Lonely Hearts Club Band";
private String artist = "The Beatles";

public void play() {
System.out.println("Playing " + title +" by " + artist);
}
}

    Component表明该类会作为组件类，并告知spring为该类创建bean

    组件扫描默认是不启用的，需要显式配置spring，从而命令它去寻找带有@Component的类，并为它创建bean

    //启动组件扫描
package com.appleieye.maya.api;

import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.Configuration;
@Configuration
@ComponentScan
public class CDPlayerConfig {
 }

    @ComponentScan注解用于启动组件扫描；如果没有特殊配置的话，@ComponentScan默认会扫描与配置类相同的包，因为CDPlayerConfig位于api这个包下，所以@ComponentScan会扫描API包以及它的子包，这样就能发现SgtPeppers，并为其创建bean。

  2. 自动装配
    自动装配就是让Spring自动满足bean依赖的一种方法，在满足依赖的 过程中，会在Spring应用上下文中寻找匹配某个bean需求的其他bean。为了 声明要进行自动装配，我们可以借助Spring的@Autowired 注解。

代码如下：

package com.appleieye.maya.api.impl;

@Component
public class CDPlayer implements MediaPlayer {

@Autowired
private CompasticDisc cd;

public void play() {
    cd.play();
}
}

spring bean装配的三种方式之Java代码配置

spring bean装配的三种方式之Java代码配置

写在前面

尽管自动化转配能够满足大部分的需求，但是有一种情况却无能为力：比如说， 你想 要将 想将第三方库中的组件装配到应用中，在这种情况下，是没有 办法在它的类上添加@Component和@Autowired注解的。这个时候，可以使用java代码或xml配置的方式。

步骤

  • 创建配置类
  • 声明简单的bean
  • 实现注入

代码如下

  1. 配置类,注意该类与自动装配的区别
package com.appleieye.maya.api;

@Configuration
public class CDPlayerConfig {

}
  1. 声明bean:要在 JavaConfig 中 声明 bean，我们需要编写一个方法， 这个方法会创建所需类型的实例，然后给这个方法添加@Bean注解。
@Bean
public CompasticDisc sgtPeppers(){
   return new SgtPeppers();
}

  1. 实现注入：将cd注入到player中去

    @Bean public CDPlayer cdPlayer( CompactDisc compactDisc) {
return new CDPlayer( compactDisc); 
}

    这是引用其他bean到最佳选择。 cdPlayer()方法请求一个CompactDisc作为参数。当Spring调用cdPlayer()创建CDPlayerbean的时候，它会自动装配一个CompactDisc到配置方法之中。然后，方法体就可以按照合适的方式来使用它。

spring依赖注入示例之hello world

写在前面

任何一个应用都会有两个或两个以上的类组成，这些类相互协作以完成特定的逻辑，这期间会出现高度耦合的代码。而依赖注入的出现，可以让我们比较容易的实现松耦合。通过依赖注入，对象的依赖关系由系统中负责协调各对象的第三方组件在创建对象的时候进行设定。对象无需创建或管理它们的依赖关系，而依赖关系会自动被注入到需要它们的对象当中去。

hello world

  • 情景：有一个类，它负责根据时间来输出hello world，如早上输出“hello world in morning”,晚上输出“hello world at night”。
  • 设计：Interface–HelloWorldService定义一个接口sayHelloWorld(),而HelloWorldA和HelloWorldB负责具体的实现；类OutPutHelper实现具体的“业务”。
  • 步骤：
    • 新建maven工程，用来管理Spring jar包依赖
    • 在pom文件中引入spring依赖
    • 代码实现如下：
        //HelloWorldService接口定义
        public interface HelloWorldService {
        String sayHelloWorld();
        }
        //HelloWorldA的实现
        public class HelloWorldA implements HelloWorldService {
        public String sayHelloWorld() {
        return "Hello World in  morning";
        }
        }
        //HelloWorldB的实现
        public class HelloWorldB implements HelloWorldService {
        public String sayHelloWorld() {
        return "Hello World in night";
        }
        }
        //输出类实现
        public class OutPutHelper {
        private HelloWorldService helloWorldService;

         public OutPutHelper(HelloWorldService  helloWorldService){
        this.helloWorldService = helloWorldService;
        }

         public void outPut(){
         System.out.println(helloWorldService.sayHelloWorld());
        }
        }
* spring配置文件

        <bean id="helloWorldA" class="com.appleieye.service.impl.HelloWorldA" />
         <bean id="helloWorldB"  class="com.appleieye.service.impl.HelloWorldB" />
        <bean id="outPutHelper" class="com.appleieye.outputer.OutPutHelper" >
        <constructor-arg ref="helloWorldA" />
        </bean>


* 测试


           public class Main {
           public static void main(String[] args) {
           ClassPathXmlApplicationContext context = new ClassPathXmlApplicationContext("spring.xml");
            OutPutHelper outPutHelper = (OutPutHelper) context.getBean("outPutHelper");
           outPutHelper.outPut();

             }
          }

最后：Spring通过xml配置的方式实现了bean类的依赖注入。只是一种注入方式，其他方式在以后的文章中也会介绍。
